Project Overview

CloudMart is an enterprise-grade e-commerce application designed to demonstrate the power of microservices architecture, containerization, and automated cloud deployments.

The system is developed in parallel by two core engineers:

  • Nischal Khanal - Leading Operations, Orchestration, and Analytics.
  • Gajananda Mani Adhikari - Leading Data, Users, Finance, and Notifications.

This documentation hub acts as our single source of truth. By strictly adhering to the API contracts defined here, Nischal and Gajananda can work at full velocity simultaneously.

Step-by-Step Parallel Workflow

1
Lock the Contracts

Both Nischal Khanal and Gajananda Mani Adhikari study this API Reference. Agree on request bodies, response shapes, and event schemas.

2
Start Docker Infrastructure

Run docker-compose up postgres rabbitmq redis -d to spin up shared data stores locally.

3
Build Against Mocks, Not Each Other

When Nischal builds Order Service, he mocks the Payment Service response. When Gajananda builds Payment Service, he tests it by manually publishing OrderCreatedEvent JSON to RabbitMQ via the management UI.

4
Integration Day

Run docker-compose up --build. Replace mocks with real local DNS (e.g. http://payment-service:3006). Run the Postman test collection.

RabbitMQ Event Architecture

The Fanout / Pub-Sub Pattern (Multiple Consumers)

CloudMart uses a Pub-Sub pattern using RabbitMQ Exchanges. A single event (like OrderCreatedEvent) is published to the cloudmart.events exchange.

Multiple independent queues are bound to this exchange. This allows Payment Service, Notification Service, and Analytics Service to ALL receive a copy of the exact same OrderCreatedEvent message simultaneously, without affecting each other.

  • If the Notification Service crashes, the Payment Service still processes the payment.
  • When Notification comes back online, it reads the backlog from its dedicated queue.
